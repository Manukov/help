### File
Каждый файл с которым работает git имеет "права на файлы" которые описываются целым числом. Это число состоит из 2-х частей:
* 1-я часть (первые 3 цифры) - это тип объекта, который сохраняется в git. 100 - это файл.
* 2-я часть (последние 3 цифры) - это [права на этот файл](#file-permission)

### File Permission
Среди всех прав на файл git хранит только право на выполнение:
* исполняемых файл - код 755
* не исполняемый файл - код 644

Изменение права на выполнение git рассматривает как изменение содержимого самого файла. Для Windows это упрощается тем, что файловая система не поддерживает отдельное право на выполнение файла и оно, как правило, объединено с правом на чтение (предполагается, что если файл можно читать, то его можно и выполнять). Git проверяет, поддерживает ли файловая система отдельное право на запуск исполняемых файлов и если не поддерживает, как в случае с windows то устанавливает настройку ```git config core.filemode false``` которая указывает git что на права в файловой системе вообще смотреть не нужно. Поэтому все файлы сохраняемые под Windows будут все 644.

### Staging Area
Staging Area (область подготовленных файлов, индекс) - это файл хранящийся в каталоге .git, который содержит информацию о том, что должно войти в следующий коммит. Область подготовленных файлов реализует двухступенчатую систему сохранения изменений, позволяющую разделить все изменения сделанные в working directory на отдельные коммиты. Например: 
* в один коммит попадают только связанные изменения, а не все изменения которые сделаны в [working directory](#working-directory). 
* выбрать из всех изменений только те, которые представляют законченную функциональность, а те которые находятся в работе, остаются в working directory; 

Для добавления файла в staging area используется команда ```git add```, но если в файл который уже ранее был добавлен в staging area внести после этого изменения, то он будет уже modified и его заново нужно добавлять в staging area.

* ```git add``` - добавление изменений в staging area (что что было красным, станет зеленым)
* ```git reset```- удаление файлов из staging area (что что было зеленым, станет красным)

### git add-command

Аргументы:
* "." это параметр, который передает в git весь текущий каталог и git добавляет в индекс все изменения сделанные в этом каталоге. 
* "-f" - это флаг, который позволяет принудительно добавлять в staging area даже файлы, которые указаны в gitignore. После добавления эти файлы далее уже будут отслеживаться git. Пример: ```git add -f .idea/project.iml``` 
* "--force"
* "-p" - это флаг, который позволяет при нескольких изменениях в разных частях одного файла, добавлять в индекс отдельно каждое изменение. Пример: ```git add -p .idea/project.html``` после чего в консоли для каждого изменения появляется список действий: 
  * y - добавлять
  * n - не добавлять
  * q
  * a
  * d
  * g
  * /
  * i
  * j
  * e
  * ? - вывести справку;


### git branch-command
* "-b" - это флаг, который позволяет одновременно выполнить команды "git branch" и "git checkout"; 
* "-d" - флаг позволяет безопасно удалить ветку. Удаление ветки произойдёт только в том случае, если было проведено слияние (merge) с текущей веткой - будет выполнена проверка, было ли произведено слияние удаляемой ветки с текущей веткой;

### git commit-command
git commit - команда создающая снимок состояния на текущий момент. Выполнение команды без параметров открывает текстовый редактор для ввода сообщения, которое является обязательным. 
Аргументы:
* "-a" - это флаг, который последовательно выполняет две команды ```git add``` (добавление всех untracked file из working directory в staging area) и ```git commit``` (все файлы добавленные предыдущей командой в staging area из staging area добавляются в repository). Пример: ```git commit -a```. В результате выполнения команды будет создан коммит. Ограничение этого флага в том, что "-a" работает только с теми файлами которые уже есть в staging area и не работает с untracked-файлами;
* "--all" - псевдоним для "-a"
* "-am" - это флаг, который эквивалентен по функциональности флагам "-a" и "-m". Пример: ```git commit -am 'Commit message''```. В результате выполнения команды будет создан коммит;
* "--amend" - это флаг, который позволяет изменять предыдущий (последний) коммит и добавлять к нему все проиндексированные изменения. Кроме того, этот флаг откроет настроенный текстовый редактор системы с предложением изменить указанный ранее комментарий к коммиту;
* "-m" - это флаг, который позволяет добавить сообщение к коммиту без вызова редактора. Пример: ```git commit -m 'Init commit''```. В результате появляется сообщение "create mode 100644 src/index.java" где 100644 это права на доступ к файлу;

git stash - команда, которая позволяет временно спрятать уже сделанные изменения, чтобы переключиться на другую ветку не делая коммит и не теряя уже сделанные изменения. При
